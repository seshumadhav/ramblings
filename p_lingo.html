Common Software Engineering Terms

Target audience: Software Engineers and leaders

Being familiar with common software engineering terms helps you quickly grasp ideas, communicate complex ideas quickly, recognize patterns or trade-offs without lengthy explanations, making you a more effective collaborator.

Here are few less common (or more tricky) terms that I repeatedly encountered and/or applied:

  • Locality of Reference Principle
    • "A program will access a particular set of data over a short period of time"
    • Temporal locality - If a memory location is accessed, it is likely to be accessed again soon.
    • Spatial locality - if a memory location is accessed, nearby memory locations are likely to be accessed soon.
  • Pareto Principle (80/20 rule)
    • "80% of effects come from 20% of causes"
    • A small portion of inputs often leads to the majority of outputs or results
  • Parkinson's Law
    • "Work expands so as to fill the time available for its completion"
  • Bloom Filters (source: how.dev)
    • A probabilistic data structure that uses bit vectors. Does not store actual data.
    • It can confirm absence of a Key accurately and quickly.
    • However presence cannot be confirmed.
  • Ephemeral (source: chatgpt.com)
    • Something short-lived or temporary.
    • Data or resources exist only for a brief period and can be destroyed/recreated without consequence.
  • Bespoke Solution
    • Built from scratch exclusively for a specific organization or client or team
  • Customized/Tailored Solution
    • Started with an off-the-shelf solution and tweaked it
  • Off the shelf solution
    • A ready-made, mass-produced product that you can buy and use as-is.
    • Eg: CRM solutions like Salesforce, Google Office Suite
  • Out of the box solution
    • Works immediately upon setup or installation, with no (or minimal) configuration needed.
    • Eg: Zoom, slack
  • Idempotency
    • An idempotent task produces the same result, no matter how many times we execute it.
    • Eg: Setting your Instagram profile picture to the same photo any number of times, does not change the profile pic.
  • Confusion Matrix
    • Theory:
      • Positive – Model thinks something needs to change
      • Negative – Model thinks no change is needed
      • True – Model was right
      • False – Model was wrong
    • Example:
      • FP (False Positive) – Model changes a correct word OR makes a wrong correction for a typo.
      • FN (False Negative) – Input ❌ but model does nothing (missed correction).
      • TP (True Positive) – Input ❌ and model corrects it correctly.
      • Input already spelled correctly ✅; Model leaves it unchanged – TN (True Negative)
      • Input already spelled correctly ✅; Model changes it to typo – FP (False Positive)
      • Input spelled incorrectly ❌; Model leaves it unchanged – FN (False Negative)
      • Input spelled incorrectly ❌; Model changes it to another typo – FP (False Positive)
      • Input spelled incorrectly ❌; Model properly corrects it – TP (True Positive)
